## 二、智能体职责与接口

### 2.1 接口定义

| 智能体 | 输入 | 输出 | 依赖工具 | 调用次数 |
|--------|------|------|----------|----------|
| **检索智能体** | 主题 | 上下文文本 + 引用列表 | Tavily Search API | 1次（除非补充搜索） |
| **生成智能体** | 主题 + 上下文 + 引用 | 论文初稿（Markdown） | OpenRouter LLM | 1次（初稿） |
| **反思智能体** | 论文版本 + 引用表 + 字数目标 | 评估报告（分数、问题列表、修改建议） | OpenRouter LLM | 每迭代1次 |
| **修订智能体** | 论文版本 + 评估报告 | 论文新版本 | OpenRouter LLM | 每迭代1次 |

### 2.2 通信机制

四个智能体均通过 MCP 协议与外部工具（搜索、LLM）交互，彼此通过**黑板（共享数据结构）**通信，避免直接消息传递的复杂性。

## 三、协调器工作流程

### 3.1 主流程

```
1. 接收主题、参数（最大迭代次数max_iter=3，目标字数1900±200）
2. 检查缓存（主题+模型 → 最终论文）。命中则直接返回。
3. 调用检索智能体，获得上下文和引用表。
4. 调用生成智能体，获得论文初稿 V0。
5. 循环 for iteration = 1 to max_iter:
   a. 调用反思智能体，输入 V(i-1)，输出评估报告。
   b. 如果评估报告中的"综合分数" >= 8.0 且 字数偏差 < 200：
        跳出循环，将 V(i-1) 作为最终论文。
   c. 否则调用修订智能体，输入 V(i-1) + 评估报告，输出 V(i)。
   d. 如果 iteration == max_iter，强制接受 V(max_iter)。
6. 将最终论文写入缓存。
7. 输出论文及迭代统计（次数、每轮分数）。
```

## 四、反思智能体详细设计

### 4.1 评估维度与分数体系

反思智能体从以下六个维度评分，每项 0–10 分，综合分数为加权平均：

| 维度 | 权重 | 检查内容 |
|------|------|----------|
| **内容准确性** | 25% | 是否基于搜索资料，有无明显事实错误 |
| **结构逻辑** | 20% | 章节连贯性、论证层次、过渡自然 |
| **引用内容恰当性** | 10% | 引用是否真正支持论点，而非随意堆砌 |
| **字数符合度** | 15% | 偏差绝对值 ≤200 字得满分，每超50字扣1分 |
| **语言表达** | 10% | 语法、用词、学术风格 |
| **格式规范性** | 10% | Markdown 语法、论文结构、引用格式（来自规则检查器） |

## 五、格式审查模块

### 5.1 审查目标范围

考虑到论文最终输出为 **Markdown 格式** 并在终端通过 `rich` 渲染，格式审查聚焦于以下可自动检查的方面：

| 审查类别 | 具体检查项 | 严重程度 |
|----------|-----------|----------|
| **Markdown 语法** | 标题标记 `#`、`##`、`###` 是否正确使用（无多余空格，层级连续） | 中 |
| | 列表标记 `-` 或 `1.` 是否前后有空行 | 低 |
| | 加粗 `**text**` 是否成对出现 | 中 |
| | 链接 `[text](url)` 格式是否正确 | 低 |
| **论文结构** | 是否包含标准章节：摘要、引言、结论（每项缺失扣分） | 高 |
| | 章节编号是否连续（如 1. 2. 3. 不应跳号） | 中 |
| | 章节标题与内容之间是否有空行 | 低 |
| **引用格式** | 引用标记 `[n]` 是否紧跟在标点符号之前（句号、逗号后） | 低 |
| | 引用编号是否连续且 ≤ 参考文献总数 | 高 |
| | 参考文献列表格式是否统一（`[n] 标题. 链接`） | 中 |
| **一致性** | 全文使用同一级别的标题风格 | 低 |
| | 图表引用是否标注（可选） | 低 |

### 5.2 集成方式

格式审查作为**反思智能体**的一个独立子模块，采用**规则+LLM 混合方案**：

- 先使用正则表达式/解析器进行确定性格式检查
- 再将结果作为结构化输入提供给 LLM 进行综合评估

**优势**：格式规则是确定的，无需依赖 LLM 判断，速度快且准确。

### 5.3 规则检查器实现

```python
import re
from typing import Dict, List, Tuple

class FormatChecker:
    def __init__(self, paper: str):
        self.paper = paper
        self.lines = paper.split('\n')
    
    def check_heading_syntax(self) -> Tuple[int, List[str]]:
        """检查标题语法，返回 (分数, 问题列表)，满分10分"""
        score = 10
        issues = []
        for i, line in enumerate(self.lines):
            if line.startswith('#'):
                if re.match(r'^#+[^ ]', line):
                    score -= 1
                    issues.append(f"行{i+1}: 标题标记后缺少空格 '{line[:20]}...'")
        return max(score, 0), issues
    
    def check_structure(self) -> Tuple[int, List[str]]:
        """检查必需章节是否存在"""
        required_sections = ['摘要', '引言', '结论']
        found = [sec for sec in required_sections if sec in self.paper]
        score = int(len(found) / len(required_sections) * 10)
        missing = [sec for sec in required_sections if sec not in found]
        issues = [f"缺少必需章节: {', '.join(missing)}"] if missing else []
        return score, issues
    
    def check_citation_format(self, citations_count: int) -> Tuple[int, List[str]]:
        """检查引用标记格式和范围"""
        refs = re.findall(r'\[(\d+)\]', self.paper)
        if not refs:
            return 0, ["论文中没有任何引用标记"]
        
        invalid_refs = [int(r) for r in refs if int(r) > citations_count or int(r) < 1]
        score = max(0, 10 - len(invalid_refs))
        issues = [f"引用编号超出范围: {invalid_refs}"] if invalid_refs else []
        
        misplaced = re.findall(r'\[\d+\]\s*[^\.,;:\s]', self.paper)
        if misplaced:
            issues.append(f"发现 {len(misplaced)} 处引用后未紧跟标点符号")
            score = max(0, score - 1)
        
        return score, issues
    
    def check_reference_list(self, citations: list) -> Tuple[int, List[str]]:
        """检查参考文献列表格式"""
        ref_section = re.search(r'(?<=参考文献\s*\n)(.*?)(?=\n\n|\Z)', self.paper, re.DOTALL)
        if not ref_section:
            return 0, ["找不到参考文献列表"]
        
        lines = ref_section.group(0).split('\n')
        expected_count = len(citations)
        actual_count = len([l for l in lines if re.match(r'^\[\d+\]', l.strip())])
        if actual_count != expected_count:
            return 5, [f"参考文献条目数不一致：正文引用{expected_count}条，列表只有{actual_count}条"]
        
        score = 10
        issues = []
        for i, line in enumerate(lines):
            if re.match(r'^\[\d+\]', line.strip()):
                if 'http' not in line and '://' not in line:
                    score -= 1
                    issues.append(f"参考文献第{i+1}条缺少链接")
        return max(score, 0), issues
    
    def full_check(self, citations: list) -> Dict:
        """执行所有检查，返回各维度分数和总问题列表"""
        heading_score, heading_issues = self.check_heading_syntax()
        structure_score, structure_issues = self.check_structure()
        citation_score, citation_issues = self.check_citation_format(len(citations))
        ref_score, ref_issues = self.check_reference_list(citations)
        
        weights = {'heading': 0.2, 'structure': 0.3, 'citation': 0.3, 'reference': 0.2}
        total_score = (heading_score * weights['heading'] + 
                       structure_score * weights['structure'] +
                       citation_score * weights['citation'] +
                       ref_score * weights['reference'])
        
        all_issues = heading_issues + structure_issues + citation_issues + ref_issues
        return {
            "format_score": round(total_score, 1),
            "sub_scores": {
                "heading": heading_score,
                "structure": structure_score,
                "citation_mark": citation_score,
                "reference_list": ref_score
            },
            "issues": all_issues[:10]
        }
```

### 5.4 集成到评估流程

1. 先运行 `FormatChecker` 获得**格式得分**和**格式问题列表**
2. 将格式问题列表**注入**到 LLM 的 Prompt 中
3. LLM 评估内容维度，格式分数直接使用规则检查器结果

| **预留** | - | 10% | - |
| **综合总分** | - | - | **6.995 ≈ 7.0** |

## 六、修订智能体详细设计

### 6.1 输入与输出

| 项目 | 说明 |
|------|------|
| **输入** | 当前论文版本 + 评估报告（JSON） |
| **输出** | 修改后的新版本论文（完整 Markdown） |

修订智能体**仅修改报告中指出的问题**，不重写全文，以保持迭代的稳定性。

### 6.2 修订智能体 Prompt 模板

```
你是一位学术论文编辑。请根据审稿意见修改论文。

【原始论文】
{current_paper}

【审稿意见】
{assessment_report}

【格式问题】
{format_issues}

修改指导原则：
1. 只修改审稿意见中明确指出的位置和问题。
2. 对于缺失引用，请在相应句子后添加正确的[n]标记（参考提供的参考资料）。
3. 对于事实错误，请根据参考资料更正数据或措辞。
4. 对于结构问题（如结论太短），请扩展对应段落，保持与全文风格一致。
5. 格式问题修复指南：
   - 标题标记后缺少空格：将 "#标题" 改为 "# 标题"
   - 引用编号超出范围：删除该引用标记，或替换为最相关的有效引用
   - 缺少参考文献条目：保持现状，不要自动生成虚假引用
6. 保持其他未提及的部分完全不变。
7. 输出完整的修改后论文（Markdown格式），不要输出解释。
```

### 7.1 触发条件

反思智能体发现当前搜索资料不足以支撑某些论点时，可设置：
- `supplementary_search_needed = true`
- `suggested_search_queries = ["查询1", "查询2"]`

### 7.2 处理流程

```python
if assessment["supplementary_search_needed"]:
    new_context = retriever.search(assessment["suggested_search_queries"])
    context += "\n\n[补充资料]\n" + new_context
    current_paper = generator.generate(topic, context, citations)  # 重新生成
    continue  # 跳过本次修订，重新进入反思
```

## 八、缓存策略

### 8.1 缓存类型

| 缓存类型 | 键 | 有效期 | 说明 |
|----------|----|--------|------|
| 搜索结果 | `search:{topic}` | 7天 | 同一主题的搜索资料可复用 |
| 最终论文 | `paper:{topic}:{model}:{max_iter}` | 永久 | 缓存最终版本，避免重复迭代 |
| 中间版本 | `paper:{topic}:{model}:iter{iteration}` | 1天 | 可选，用于分析迭代效果 |

### 8.2 注意事项

- 反思智能体的评估结果**不缓存**，因为每次迭代的论文版本不同
- 缓存键包含模型名称，不同模型的生成结果分开存储

## 九、迭代终止条件

### 9.1 终止条件列表

| 条件 | 说明 |
|------|------|
| **分数达标** | 综合分数 >= 8.0 |
| **字数达标** | 字数偏差 < 200 |
| **改进停滞** | 连续两次迭代综合分数提升 < 0.2 |
| **最大迭代** | 达到 max_iter（默认3次） |
| **用户中断** | CLI 支持 Ctrl+C 并保存当前版本 |

```bash
# 默认启用反思，最多迭代3次
papergen generate "碳中和" --reflect

# 关闭反思（线性模式，更快）
papergen generate "碳中和" --no-reflect

# 显示每轮评估报告
papergen generate "碳中和" --verbose

# 禁用格式检查（调试用）
papergen generate "碳中和" --no-format-check
```

## 十一、实现注意事项

### 11.1 Token 消耗控制

| 阶段 | Token 消耗 | 说明 |
|------|-----------|------|
| 反思 Prompt | 约 2000–3000 tokens | 包含当前论文全文 |
| 修订 Prompt | 约 2000–3000 tokens | 包含当前论文全文 |
| 单次迭代 | 约 4000–6000 tokens | 反思 + 修订 |
| 3次迭代总计 | 约 $0.15–0.30 | 使用 Claude 3.5 Sonnet |

### 11.2 状态持久化

协调器需要将每次迭代的论文版本保存到临时文件，以防中途崩溃导致进度丢失。

### 11.3 引用一致性

修订智能体修改段落时，必须确保引用编号与参考文献表保持对应。建议在修订后调用后处理函数验证。

### 11.4 字数微调

反思智能体会报告字数偏差，修订智能体应优先调整长度（例如通过扩写/精简结论段），而不是改变核心内容。

## 十三、方案对比总结

### 13.1 线性方案 vs 反思智能体方案

| 特性 | 线性方案 | 反思智能体方案 |
|------|----------|----------------|
| **输出质量稳定性** | 中（依赖单次生成） | 高（通过迭代修复问题） |
| **平均耗时** | 20秒 | 40–60秒（2–3次迭代） |
| **平均成本** | $0.05 | $0.12–0.20 |
| **可解释性** | 低（黑盒生成） | 高（每轮评估报告可见） |
| **适合场景** | 快速原型、批量生成 | 正式报告、学术论文 |

### 12.2 结论

反思智能体方案显著提升输出质量（特别是引用完整性和事实准确性），成本增加可控，适合对质量要求较高的论文生成 Demo。

**建议**：
- 默认启用反思模式
- 保留 `--no-reflect` 选项供快速测试使用
- 格式审查作为必选模块，提升论文规范性

## 附录：扩展性

未来可以增加的功能：

| 功能 | 说明 |
|------|------|
| 图表引用一致性 | 检查是否都有 `![alt]` 标签 |
| 参考文献 DOI 验证 | 验证 DOI 或 ISBN 格式 |
| 引用去重检测 | 同一来源多次引用是否使用同一编号 |
| 段落均匀性 | 字数分布的段落均匀性检查 |

所有扩展都可以在 `FormatChecker` 类中逐步添加方法，保持反思智能体的核心 LLM 调用不受影响。
